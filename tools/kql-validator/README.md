# KQL Validator

Validates `.kql` files generated by the `kql-builder` agent for structural correctness, common LLM errors, and pipeline conventions. Runs as a step in the `/kql` command pipeline (Step 4b) between query generation and reviewer pass.

## Usage

```bash
# Validate a single file
python tools/kql-validator/validate.py path/to/file.kql

# Validate all .kql files in a directory
python tools/kql-validator/validate.py "Completed TRR Reports/TRR0016/win/kql/*.kql"

# JSON output (for pipeline consumption)
python tools/kql-validator/validate.py --format json path/to/file.kql

# Text output (default, for humans)
python tools/kql-validator/validate.py --format text path/to/file.kql
```

**Exit codes:** 0 = all files pass, 1 = warnings only, 2 = errors found.

**Dependencies:** Python 3.10+ standard library only. No external packages.

## What Level 1 Checks

### Balanced Delimiters
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-DELIM-001` | error | Unclosed parenthesis |
| `L1-DELIM-002` | error | Unmatched closing parenthesis |
| `L1-DELIM-003` | error | Unclosed bracket |
| `L1-DELIM-004` | error | Unmatched closing bracket |
| `L1-DELIM-005` | error | Unclosed string literal |

### SQL Keyword Contamination
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-KEYWORD-001` | error | SQL keywords used instead of KQL (`SELECT`, `FROM`, `WHERE`, `LIKE`, `GROUP BY`, `ORDER BY`) |
| `L1-KEYWORD-002` | error | SQL logical operators (`AND`, `OR`, `NOT`) instead of KQL lowercase equivalents |

### Function Errors
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-FUNC-001` | error | `ago()` called with a quoted string instead of a timespan literal |

### Statement Structure
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-STRUCT-001` | error | Trailing pipe with no operator following it |
| `L1-STRUCT-002` | warning | Query does not end with semicolon |

### Pipe Structure
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-PIPE-001` | error | KQL operator without preceding pipe (`|`) or table name |

### Assignment vs. Comparison
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-EQUAL-001` | error | Single `=` in a `where` clause (assignment instead of `==` comparison) |

### Table Names
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-TABLE-001` | warning | Table name not in the known Sentinel/Defender table list |

### Time Field Convention
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-TIME-001` | warning | Wrong time field for the table platform (e.g., `Timestamp` on a Sentinel-only table, `TimeGenerated` on a Defender-only table) |

### Header Comment Convention
| Code | Severity | What It Catches |
|------|----------|-----------------|
| `L1-HEADER-001` | warning | File does not start with a header comment |
| `L1-HEADER-002` | warning | Header comment does not reference a TRR procedure ID (e.g., `TRR0011.AD.A`) |
| `L1-HEADER-003` | warning | Header comment does not reference a DDM node (e.g., `n6 "Operation Name"`) |

## Output Format

**Text (default):**
```
trr0016_win_a.kql: FAIL (2 errors, 1 warning)

  ERROR  L5:C23   L1-DELIM-001  Unclosed parenthesis opened at line 5, column 23 -- Add closing parenthesis
  ERROR  L8:C1    L1-KEYWORD-002  SQL logical operator 'AND' used instead of KQL 'and' -- Replace 'AND' with 'and'
  WARN   L1:C1    L1-TABLE-001  Table 'CustomTable_CL' not in known table list -- Verify table exists in your environment.
```

**JSON:**
```json
[
  {
    "file": "trr0016_win_a.kql",
    "level": 1,
    "status": "FAIL",
    "errors": [
      {
        "line": 5,
        "column": 23,
        "severity": "error",
        "code": "L1-DELIM-001",
        "message": "Unclosed parenthesis opened at line 5, column 23",
        "suggestion": "Add closing parenthesis"
      }
    ],
    "warnings": []
  }
]
```

## Known Tables

The validator recognizes tables from both platforms. Unknown tables produce a warning (not an error) — they might be custom tables in your environment.

**Sentinel tables:** SecurityEvent, SysmonEvent, Event, W3CIISLog, SigninLogs, AuditLogs, AADNonInteractiveUserSignInLogs, CommonSecurityLog, Syslog, Usage, Heartbeat, Perf, IdentityLogonEvents, IdentityDirectoryEvents, IdentityQueryEvents.

**Defender tables:** DeviceProcessEvents, DeviceFileEvents, DeviceRegistryEvents, DeviceImageLoadEvents, DeviceNetworkEvents, DeviceLogonEvents, DeviceEvents, IdentityLogonEvents, IdentityDirectoryEvents, IdentityQueryEvents, AlertInfo, AlertEvidence.

To add tables, edit `kql_tables.py`.

## Future Upgrade Path

### Level 2 — Schema Validation

**Implement when:** `kql-environment-profile.md` at the repo root is populated with Available Log Sources (§2) and Field Name Mappings (§3).

**What it adds:**
- Promotes `L1-TABLE-001` warnings to errors if the table isn't declared in the profile
- Validates field names in `where`, `project`, `extend`, `summarize` clauses against declared schema
- Flags field names that exist under a different name in the environment (using §3 mappings)
- Checks that telemetry sources marked "No" in §2 don't have queries generated for them

**Dependencies:** Populated environment profile. No external services (fully offline).

### Level 3 — Live Workspace Compilation

**Implement when:** Azure Log Analytics workspace is accessible with API credentials.

**What it adds:**
- Submits each query to the Log Analytics Query API with `prefer: wait=0` for compilation-only validation
- Catches errors that static analysis can't: missing workspace functions, permission issues, table schema version mismatches, custom function references

**Dependencies:** Azure subscription, Log Analytics workspace, API credentials (service principal or Azure CLI login), network access.

### How Levels Compose

Each level is a separate Python module. The `validate.py` entry point runs whichever levels are available:

- Level 1 always runs (no dependencies)
- Level 2 runs if the environment profile is populated
- Level 3 runs if Azure credentials are configured

Levels are strictly additive — each catches a different error class. Level 1 gives better error messages for structural problems even when Level 3 is available ("unclosed parenthesis at L5:C23" vs. "syntax error at line 5").
